I'm not sure about Christer's solution, but the common (Moller/Haines) Ray/Triangle collision algorithm returns a "t" parameter that represents when a collision did (or will) occur.

It considers a collision to be when t>=0 and t<=1. So "t" represents where on the motion vector (ray) the collision occurred, with t==0 being the starting end and t==1 being the end point of the ray.

If the infinite line in the direction of the ray is going to hit in the future, then t>1. If it is behind then t<0. A miss is denoted with a separate boolean (at least in my own version).

So, for an object to "miss" because you are moving too fast you'd have something like:

Frame 1) t==3.0;
Frame 2) t==1.5;
Frame 3) t==-.5;

So, instead of just testing if (t>=0 and t<=1) to denote a collision, test to see if the (0-1) range exists inside the delta since the last frame.

bool bMissed = (lastt > 1 && t < 0);

Quote:
Solutions such as Christers AABB approach are also useful - it's probably just sweeping out the corners, with an optimization to reduce it from 8 that others have noted here.
The swept separating axis test for two AABBs doesn't involve sweeping of corners (not explicitly, at least), and it also doesn't make use of ray-triangle tests in any form.

@The OP: The easiest way to think of the swept AABB-AABB test is probably as a single ray-AABB test (the ray-AABB test is well documented, and you should be able to find some code examples in the forum archives here). To express the swept AABB-AABB test as a ray-AABB test:

1. Adjust the velocities so that one of the boxes is stationary.

2. Expand the stationary box by the extents of the moving box.

3. Intersect the relative velocity vector with this expanded box.

4. If there's an intersection, the computed parametric value is the parametric time of intersection.

You can also express the problem as intersecting moving 1-d intervals, once for each cardinal axis. However you express the problem, it should also be possible to extract additional information, such as the collision normal and on which sides of the boxes the collision occurs.

Both the gamasutra article and Christer's book use the 'swept interval' form, and in each case the time of intersection (if any) is returned as an 'out' parameter via reference. Naturally with published code there's always the possibility of error, but I'd say it's fairly likely that both of these code samples are correct (if you're unsure, you could always check the errata for Christer's book).

Since the algorithm is fairly simple, perhaps you could just post your code. SAT implementations can be kind of tedious to try and debug by reading code, but maybe someone will be able to spot what you're doing wrong.

Also (and I think this has already been suggested), I suggest setting up a simple test case for which you know what the outcome should be, and then stepping through the code in the debugger to determine where the actual results are deviating from the expected results.

Ok, I see that since it is Axis aligned there is no need for dot product or cross product.

Ray - Box Intersection

We can use a box both as an object and a bounding volume. Kay and Kayjia developed a method based on "slabs" where a slab is the space between two parallel planes. So the intersection of a set of slabs defines a bounding volume or a box.

The method looks at the intersection of each pair of slabs by the ray. It finds tfar and tnear for each pair of slabs. If the overall largest tnear value i.e,. intersection with the near slab, is greater than the smallest tfar value (intersection with far slab) then the ray misses the box, else it hits the box. We can use as the simplest bounding volume the intersection of the sets of two parallel planes where each set has normals along the X, Y, or Z axes.

http://www.siggraph....ce/rtinter3.htm

jyk wrote: "Since the algorithm is fairly simple, perhaps you could just post your code."

Here it is (C#). Note that the method is a non-static member of a BoundingBox class, so it takes just a single BoundingBox argument:

public bool IntersectsMovingBox(BoundingBox _otherBox, out float _distance) {
_distance = 0f;

if (IntersectsBox(_otherBox))
return true; // Already overlapping.

Vector distDiff = (_otherBox.DistanceDelta - distanceDelta); // Treat this box as stationary.
float farthest = 1f;

for (int i = 0; i < 3; ++i) {
if (distDiff.Axis(i) < 0f) {
if (_otherBox.maxFormer.Axis(i) < minFormer.Axis(i))
return false;
if (maxFormer.Axis(i) < _otherBox.minFormer.Axis(i))
_distance = Math.Max((maxFormer.Axis(i) - _otherBox.minFormer.Axis(i)) / distDiff.Axis(i), _distance);
if (_otherBox.maxFormer.Axis(i) > minFormer.Axis(i))
farthest = Math.Min((minFormer.Axis(i) - _otherBox.maxFormer.Axis(i)) / distDiff.Axis(i), farthest);
} else if (distDiff.Axis(i) > 0f) {
if (_otherBox.minFormer.Axis(i) > maxFormer.Axis(i))
return false;
if (_otherBox.maxFormer.Axis(i) < minFormer.Axis(i))
_distance = Math.Max((minFormer.Axis(i) - _otherBox.maxFormer.Axis(i)) / distDiff.Axis(i), _distance);
if (maxFormer.Axis(i) > _otherBox.minFormer.Axis(i))
farthest = Math.Min((maxFormer.Axis(i) - _otherBox.minFormer.Axis(i)) / distDiff.Axis(i), farthest);
}

if (_distance > farthest)
return false;
}

return (_distance > 0f);
}


The minFormer and maxFormer fields denote the min and max points on the bounding box in the previous frame.

distanceDelta refers to the vector indicating how far a bounding box has moved since the previous frame.

The .Axis(index) property simply corresponds to the float value of one axis of a vector (0=x, 1=y, 2=z).

S is the set of all possible separating axes, D is an axis (or direction) in the set. W is the relative velocity of object C1 to C0 (so you treat as if C0 is stationary, and this enables you to easily get a paramaterized time for the collision. The V(i) are the verticies of the objects, so when you dot them with the D's you are finding the max and min verts in the direction of your separating axes for the two objects, and then see if the intervals overlap.

Section 5 of the MethodOfSeparatingAxes.pdf explains which axes you need to test. For AABB's the set simplifies down to something like the x,y and z axes, the velocity direction, and the velocity direction crossed with x,y and z respectively.

It's been a while since I implemented this stuff in a game, and I am going by memory, but I think that's correct. I know the documentation and code I provided you in the earlier links is solid, so just keep re-reading it and looking at/running the code until it clicks.
